1.2 File Structure

The theme follows a clean, organized structure that separates concerns and makes the codebase maintainable:

  • üìÑ functions.php: Only initialization of the autoloader and class instantiation. No business logic.
  • üìÑ style.css: Only theme metadata (Name, Version). Do not add site styles here.
  • üìÅ
    inc/ (Core Logic):
    • üìÑ Autoloader.php: Automatic class loading.
    • üìÑ Setup.php: add_theme_support, menu registration, etc.
    • üìÑ Assets.php: wp_enqueue_scripts logic (asset loading, styles, scripts).
    • üìÑ Shortcodes: Shortcode registration and functionality.
    • üìÑ Utils: Utility and helper functions.
  • üìÅ template-parts/: Reusable template components and partials.
  • üìÅ templates/: Page templates.
  • üìÅ template-parts/flexible/: Modules for ACF Flexible Content.
  • üìÑ header, footer: Global template parts.

1.3 Class Standards

All functions must be inside classes, not just placed anywhere. This ensures better code organization, prevents namespace conflicts, and makes the codebase more maintainable and testable.

  • ‚úì We use PHP Namespaces.
  • ‚úì One class = One file.
  • ‚úì Logic is split by purpose (don't write everything in one Theme.php class).
  • ‚úì Use final class for classes that should not be extended.
  • ‚úì Use static methods to avoid creating instances. This reduces memory usage and improves performance by allowing direct method calls without object instantiation.
  • ‚úì Use proper visibility modifiers (public, private, protected).

2.1 Flexible Content Strategy

We build pages like a constructor (Page Builder).

  • ‚úì Architecture: 1 Layout in ACF = 1 file in template-parts/flexible/
  • ‚úì Template Loop: We use the standard while (have_rows(...)) loop in the templates/flexible.php file.
Example: Flexible Content Loop 
<?php if (have_rows('content')): ?>
  <?php while (have_rows('content')): the_row(); ?>
    <?php get_template_part('template-parts/flexible/' . get_row_layout()); ?>
  <?php endwhile; ?>
<?php endif; ?>

  • ‚úì Image: Image ID (for full control over the img tag).

2.4 Spacer Component

For flexible content, add a Spacer component. This component allows you to create responsive spacing between sections with different values for desktop, tablet, and mobile devices. It's useful for fine-tuning the vertical rhythm of the page and ensuring consistent spacing across different screen sizes without hardcoding values in CSS.

Example: Spacer Template 
<?php 
$desktop = get_sub_field('desktop'); 
$tablet = get_sub_field('tablet'); 
$mobile = get_sub_field('mobile'); 
?>
<div class="spacer" 
     style="--desktop: <?php echo $desktop; ?>px; 
            --tablet: <?php echo $tablet; ?>px; 
            --mobile: <?php echo $mobile; ?>px">
</div>
Example: Spacer CSS 
.spacer {
  --desktop: 90px;
  --tablet: 60px;
  --mobile: 30px;
  height: var(--desktop);
  
  @media (max-width: 991.98px) {
    height: var(--tablet);
  }
  
  @media (max-width: 575.98px) {
    height: var(--mobile);
  }
}

3.1 Sanitization & Escaping

Always sanitize input and escape output. Never trust user data.

Example: Sanitization & Escaping 
// Sanitize input
$email = sanitize_email($_POST['email']);
$name = sanitize_text_field($_POST['name']);
$url = esc_url_raw($_POST['website']);
$textarea = sanitize_textarea_field($_POST['message']);

// Escape output
echo esc_html($user_input);
echo esc_attr($attribute_value);
echo esc_url($link_url);
echo esc_js($javascript_value);
echo wp_kses_post($html_content); // Allow safe HTML

3.2 Database Queries

  • ‚úó Forbidden: query_posts() (never use).
  • ‚úì Allowed: new WP_Query() or get_posts().
  • ‚úì Hooks: To modify the main query (e.g., on archive pages) use the pre_get_posts hook, don't write a new query in the template.

3.3 Nonces

  • ‚úì All forms and AJAX requests must be protected using wp_create_nonce().

5.1 Checklist

Checklist before project handover.

  • ‚úì Debug: Ensure WP_DEBUG is set to false in wp-config.php.
  • ‚úì Search Engines: Enable indexing ("Settings ‚Üí Reading ‚Üí Discourage search engines" ‚Äî uncheck the box).
  • ‚úì Permalinks: Re-save permalink settings (common 404 error fix).
  • ‚úì Admin Email: Change administrator email to client's email.
  • ‚úì Forms: Send test submission from all forms.
  • ‚úì Dev Plugins: Remove or disable all dev-only plugins (Query Monitor, Show Current Template, etc.).
  • ‚úì Security: Remove test data, check file permissions, ensure no debug information is exposed.
  • ‚úì Performance: Clear cache, optimize database, check page load speed.
  • ‚úì SSL: Verify SSL certificate is properly configured and working.
  • ‚úì Testing: Test on different devices and browsers, check for broken links (404 errors).
  • ‚úì Backup: Create a full backup of the site before going live.

7.1 Updates

Plugins, themes, and WordPress core must be kept up to date.

7.2 Backup Requirements

  • ‚úì Updates to any site components must be performed only after a full site backup. Create a complete backup (files and database) before any updates.
  • ‚úì Hosting-level backups: Backups must be performed at the hosting level, not through WordPress plugins. Hosting-level backups are more reliable, faster, and don't consume server resources or database space. They also provide better recovery options and are independent of WordPress functionality.
